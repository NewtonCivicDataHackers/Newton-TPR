# TPR Structured Data Extraction

This skill governs how the plain-text regulation sections in `sections/*.txt`
(produced from the city's PDF by `scripts/process-tpr.py`) are turned into
structured TSV datasets in `extracted_data/`.

Division of labor: **agents do the reading, Python does the checking.**
Extraction is LLM work guided by per-section specs; every result must pass the
deterministic validator before it is written back.

## Layout

```
.claude/skills/tpr-extract/
├── SKILL.md            this file — principles, procedure, conventions
├── manifest.json       one entry per TPR section: slug, status, file paths,
│                       entry-counting rules (the join table for everything)
├── schemas/
│   ├── _common.json    shared vocabulary ($defs referenced as "common#name")
│   └── <slug>.json     one schema per dataset
├── specs/
│   └── <slug>.md       per-section extraction notes: grammar, edge cases,
│                       worked examples, judgment calls
├── scripts/
│   ├── validate.py     deterministic validator (stdlib only)
│   ├── expand_dittos.py  pre-normalization (ditto marks → explicit text)
│   └── gen_readme.py   regenerates extracted_data/README.md from schemas
└── parsers/           deterministic re-derivations used as a SANITY CHECK,
                       not as the source of truth — see parsers/README.md
```

Naming: each section gets a **slug declared once in manifest.json** (e.g.
`147` → `stop_signs`). The slug names the schema, the spec, and the output
file `extracted_data/<section>_<slug>.tsv`. Slugs are stable identifiers —
they do NOT change when the city reword a section title. Source files stay
bare-numbered (`sections/147.txt`); those names are generated by
`process-tpr.py` and referenced from `index.json`.

## Output format: TSV

- Tab-separated, UTF-8, one header row, one row per **atomic regulation**
  (one sign, one street segment, one restriction clause).
- Fields may not contain literal tabs or newlines. Collapse all internal
  whitespace to single spaces (the PDF text wraps lines mid-clause).
- ASCII-normalize punctuation: curly quotes/apostrophes (’ ‘ ” “) and prime
  marks (′ ″) become straight ASCII (' "); this includes feet marks
  (`74'`). The source PDF uses curly glyphs inconsistently; normalizing makes
  `source_text` keys stable and downstream string-matching/geocoding reliable.
  `source_text` is otherwise verbatim (whitespace-collapsed).
- No quoting, no escaping — ever. If a value seems to need it, the value is
  wrong.
- Column order must match the schema's `columns` array exactly.

## Conventions

1. **Provenance is mandatory.** Every row carries `source_text`: the verbatim
   source clause, whitespace-collapsed. This is what makes hallucination and
   omission detectable. `source_text` values must be unique within a file.
2. **Never guess.** If the source text is ambiguous, leave the structured
   field empty, keep the information in `location_detail`/`notes`, and record
   the pattern in the section's spec file. An empty cell is honest; a guessed
   one poisons the dataset.
3. **Vocabulary is declared once.** Directions, sides, time ranges, day sets
   live in `schemas/_common.json` and are referenced from dataset schemas as
   `"$ref": "common#direction"`. To use a new value, add it to `_common.json`
   first — never inline it in one dataset.
4. **Specs accumulate knowledge.** Every quirk discovered during extraction
   (splitter artifacts, unclosed parentheses, grammar variants) gets written
   into `specs/<slug>.md` so the next extraction inherits it.
5. **Docs are generated.** Never hand-edit `extracted_data/README.md`; run
   `scripts/gen_readme.py`.
6. **Design for geocoding.** The downstream goal is to geocode every
   location. Location atoms must land in their own normalized columns —
   `street`, `cross_street`, segment endpoints (`from_point`/`to_point`),
   `side`, numeric offsets/lengths with units in the column name
   (`offset_feet`, `length_miles`) — never buried in a free-text cell when
   the grammar provides them. An intersection row should geocode from
   `street` + `cross_street` alone; a segment row from `street` +
   endpoints.

## Pre-normalization (for sections with source shorthand)

Some sections encode entries with shorthand that is risky to interpret during
structured extraction. Resolve it FIRST with a deterministic, verifiable pass
that emits explicit text in the same shape as the source, then extract from
the expanded text. Keep passes separate so each is checkable on its own:
mechanical normalization vs. interpretive structuring.

- **Ditto marks** (section 84 speed zones): `0.52 “ “ 30 “ “ “` repeats the
  tokens above it. `scripts/expand_dittos.py` expands each ditto mark
  positionally from the nearest non-ditto reference line, touching only
  ditto lines (everything else byte-identical) and self-verifying that the
  result is a complete phrase. Run it, eyeball the intermediate, then do
  pass 2 (structuring, e.g. double-extraction) over the expanded text — the
  two readers can then only diverge on structure, not on ditto reading.

This generalizes to other shorthand (220 em-dash inline forms; 176 numbered
sub-clauses inheriting a street heading): expand shorthand → explicit text →
structure.

## Procedure: extracting or refreshing a dataset

1. Read `manifest.json`; find the section's entry. Read its **spec** and
   **schema** (never extract without them) plus `schemas/_common.json`.
2. Read the source `sections/<n>.txt` in full.
3. Extract every entry to rows per the schema. Apply the conventions above.
4. Write `extracted_data/<section>_<slug>.tsv` and update the row count and
   `revision_date` in `extracted_data/extraction_manifest.json` (the
   `revision_date` must equal `index.json`'s).
5. Run `python3 .claude/skills/tpr-extract/scripts/validate.py` and fix
   until it passes. Do not loosen counting rules to make a failure go away —
   a count mismatch means dropped or invented rows until proven otherwise.
6. Run `scripts/gen_readme.py`.
7. If you learned anything about the section's grammar, update its spec.

When a **new TPR revision** lands, refresh every `active` dataset whose
section text changed (`git diff` on `sections/` shows which) and bump
`extraction_manifest.json`'s `revision_date`. The validator fails on
revision mismatch, so a half-done refresh cannot pass silently.

## Adding coverage for a new section

1. Read the section text; describe its grammar in `specs/<slug>.md` with
   2–3 worked examples, including at least one awkward entry.
2. Write `schemas/<slug>.json` reusing `common#` defs for every field that
   has one.
3. In `manifest.json`: set `status: "active"` and write the `counting` rules
   (see below). Then follow the procedure above.

Sections that are blanket rules with no enumerable entries (e.g. 168's
citywide 3-hour limit) are `status: "excluded"` with a `reason` — coverage
gaps must be decisions, not accidents.

## Manifest counting rules

`counting` makes completeness checkable without parsing:

```json
"counting": {
  "region_start": "following locations:",   regex; entries begin after match
  "region_end": "\\(Rev\\. Ords\\.",        regex; entries end before match
  "entry_start": "^[A-Z]"                   regex; a paragraph matching this
}                                            is an entry, otherwise it is a
                                             continuation of the previous one
```

The validator splits the region into blank-line-separated paragraphs and
counts those matching `entry_start`; the TSV must have exactly that many
rows. Continuation paragraphs (PDF artifacts that split an entry) start
lowercase and are not counted.

When a section's completeness cannot be checked by a single regex — mixed
entry formats, or rows recovered from a PDF page because text extraction
scrambled a table — set `"counting": {"verified_by": "<how>"}` instead.
`validate.py` then skips the regex count for that dataset; completeness must
be established by other documented means (e.g. section 84: a per-stanza
distance checksum for prose stanzas, plus column-multiset reconciliation for
PDF-read tables — both described in the spec).

Add `"unit": "line"` when entries are one-per-line with no blank lines
between them (e.g. district membership lists); the validator then counts
non-blank lines matching `entry_start` instead of paragraphs. Default is
`"paragraph"`.

## Schema format

Plain JSON, validated by `validate.py` (stdlib subset, not full JSON Schema):

```json
{
  "title": "Yield sign locations",
  "columns": ["street", "direction", "...", "source_text"],
  "fields": {
    "street":      {"description": "...", "required": true},
    "direction":   {"description": "...", "$ref": "common#direction"},
    "source_text": {"description": "...", "required": true}
  },
  "unique": ["source_text"]
}
```

Field keys: `required` (bool), `type` (`string`|`integer`|`number`),
`enum`, `pattern` (full-match), `$ref` (`common#<def>`); local keys override
the ref'd def. Empty cells are legal iff `required` is false.

## Double-extraction agreement test

Before marking a newly-activated dataset done, have a second, independent
agent (a different model is fine and proves portability — Sonnet works)
blind-extract the same section from only the skill files and source text,
writing to a temp path, never reading `extracted_data/`. Diff the two
extractions cell-by-cell keyed on `source_text`:

- `source_text` sets must match exactly (else someone dropped/invented rows).
- Every differing cell is a **spec gap**: settle the interpretation, encode it
  in the spec as a field-placement rule or a gold row, and re-run until the
  extractions are identical.

Section 148 calibration: first blind Sonnet run was schema-valid with 21/32
rows identical and every divergence a spec gap; after one tightening round,
32/32 byte-identical. Spec maturity, not model choice, determines
consistency.

## Deterministic parsers (`parsers/`) — a sanity check, not an authority

A few sections have a regular-enough grammar to re-derive deterministically
from the section text (84 stanzas, 177, 220). Those parsers live in
`parsers/` and `parsers/check.py` diffs them against the committed TSVs.

Be clear about their role — and do NOT repeat the tempting mistake of treating
"promote to a deterministic parser" as a stable, trustworthy endpoint:

- The TPR is **human-edited**. Every revision can introduce a new phrasing,
  structure, typo, or inconsistency that a parser frozen against today's
  formats won't handle — possibly *silently*. So the LLM, which interprets
  meaning, is the more format-robust extractor; the parser is the brittle one.
- Therefore the parser's value is as a cheap, independent **second opinion /
  tripwire**, not as a source of truth. The committed `extracted_data/*.tsv`
  remain authoritative. On a new revision, run `parsers/check.py`: agreement
  (plus the method-independent verifiers in `validate.py`) is strong evidence a
  section didn't drift; **divergence or a parser failure is a signal for human
  review**, never automatic correction.
- This is one leg of a triangulation — LLM extraction + deterministic parser +
  `validate.py`'s checksums/counts/multiset reconciliation. Where they
  disagree is exactly what a person should look at. Note the residual: silent
  semantic drift that still *looks* valid (e.g. a coordinate typo) is caught by
  none of these — only by a human reading the per-revision diff.
